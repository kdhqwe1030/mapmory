# Mapmory - Step 2 기획 정리

## 1. 단계 목표

1단계에서 지도 기반 기본 기능(지도 표시, 장소 검색, 장소 선택)이 완료되었다고 가정한다.  
2단계에서는 사용자가 장소를 단순히 검색하는 수준을 넘어, **직접 만든 카테고리 안에 장소를 저장하고 관리할 수 있는 구조**를 만든다.

이 단계의 핵심은 다음과 같다.

- 사용자가 직접 카테고리를 생성할 수 있다.
- 장소를 특정 카테고리에 저장할 수 있다.
- 각 장소의 방문 여부를 관리할 수 있다.
- 방문했다면 방문 날짜를 기록할 수 있다.
- 지도와 별개로 관리되는 TODO 리스트 기능을 추가한다.
- TODO와 장소를 연결할 수 있도록 한다.
- 지도 화면과 관리 화면 사이를 자연스럽게 이동할 수 있는 진입 구조를 만든다.

---

## 2. 핵심 컨셉

이번 단계에서 Mapmory는 단순한 지도 서비스가 아니라,  
**장소 저장 + 방문 관리 + 할 일 연결**이 가능한 개인 아카이브 서비스로 확장된다.

즉, 사용자는 다음과 같은 흐름으로 서비스를 이용할 수 있다.

1. 장소를 검색한다.
2. 원하는 장소를 저장한다.
3. 저장할 때 특정 카테고리를 선택한다.
4. 필요하다면 새 카테고리를 직접 만든다.
5. 장소별 방문 여부를 관리한다.
6. 방문 완료 시 날짜를 기록한다.
7. TODO를 별도로 관리한다.
8. 특정 장소와 특정 TODO를 연결한다.

---

## 3. 주요 기능

### 3-1. 사용자 정의 카테고리 기능

사용자는 자신만의 카테고리를 생성할 수 있다.

예시:

- 카페
- 대형카페
- 숙소
- 액티비티
- 야경 명소
- 데이트 코스

카테고리는 사용자가 자유롭게 만들 수 있으며,  
각 장소는 하나 이상의 카테고리에 저장될 수 있도록 설계하는 것도 고려할 수 있다.

기본 MVP에서는 우선 다음 두 방식 중 하나를 선택한다.

#### 방식 A. 장소는 하나의 카테고리에만 속함

- 구현이 단순하다.
- 초반 구조를 빠르게 안정화할 수 있다.

#### 방식 B. 장소는 여러 카테고리에 동시에 속할 수 있음

- 확장성이 높다.
- “카페”이면서 동시에 “데이트코스”일 수 있다.

초기 단계에서는 **구현 난이도를 낮추기 위해 A 방식(단일 카테고리)** 로 시작하고,  
추후 다중 카테고리 구조로 확장하는 방향이 적절하다.

---

### 3-2. 장소 저장 기능 확장

사용자가 장소를 선택하면 단순 저장이 아니라,  
**어떤 카테고리에 저장할지 선택하는 단계**를 거친다.

예시 흐름:

장소 선택  
→ 저장하기 클릭  
→ 저장할 카테고리 선택  
→ 없으면 새 카테고리 생성  
→ 저장 완료

예시:

- “Cafe Onion” → 카페 카테고리에 저장
- “파르나스 호텔” → 숙소 카테고리에 저장

---

### 3-3. 방문 상태 관리

각 저장된 장소는 방문 상태를 가져야 한다.

상태 예시:

- 방문 전
- 방문 완료

영문 상태 예시:

- Not Visited
- Visited

방문 완료로 변경하면 다음 정보를 추가로 기록할 수 있다.

- 방문 날짜
- 선택적으로 간단한 메모 (추후 확장 가능)

예시:

- Cafe Onion / 방문 완료 / 2026-03-20
- 파르나스 호텔 / 방문 전

---

### 3-4. 방문 날짜 기록

방문 완료 상태인 장소는 날짜를 반드시 또는 선택적으로 기록할 수 있어야 한다.

활용 방식:

- 카테고리 내 장소 목록에서 방문 날짜 확인
- 향후 Timeline / Calendar View 확장 시 데이터 재사용 가능
- “언제 어디를 갔는지”를 시간 순으로 나열하는 기능의 기반 데이터가 됨

즉, 이 단계의 방문 날짜 데이터는 이후 확장 기능의 핵심이 된다.

---

### 3-5. TODO 리스트 기능

지도와 직접적으로 연결되지 않는 **독립적인 TODO 섹션**을 추가한다.

이 TODO는 “오늘 해야 할 일” 수준보다는,  
데이트와 관련된 준비 항목 / 계획 항목 / 체크리스트 개념에 가깝다.

예시:

- 예약하기
- 사진 찍기
- 근처 맛집 찾기
- 케이크 주문하기
- 체크인 시간 확인하기

TODO는 장소와 별개로 생성하고 관리할 수 있어야 한다.

주요 기능:

- TODO 생성
- TODO 수정
- TODO 완료 처리
- TODO 삭제

---

### 3-6. 장소와 TODO 연결

TODO는 특정 장소와 연결할 수 있어야 한다.

예시:

- “파르나스 호텔” ↔ “체크인 시간 확인하기”
- “한강공원” ↔ “돗자리 챙기기”
- “성수 카페” ↔ “예약 가능한지 확인하기”

이를 통해 장소를 단순 저장하는 것을 넘어서,  
**장소와 행동 계획이 연결된 구조**를 만들 수 있다.

연결 방식 예시:

- 장소 상세 화면에서 관련 TODO 추가
- TODO 생성 시 연관 장소 선택
- TODO 목록에서 연결된 장소 확인 가능

---

## 4. 화면 구조

## 4-1. 지도 화면

지도 화면은 여전히 서비스의 중심 진입점이다.

구성:

- 지도
- 검색바
- 장소 마커
- 장소 상세 Bottom Sheet
- 우측 하단 Floating Action Button

지도 화면에서 장소를 검색하고 저장할 수 있어야 하며,  
저장 시 카테고리 선택 UI가 자연스럽게 이어져야 한다.

### 장소 저장 흐름

1. 지도에서 장소 검색
2. 장소 상세 Bottom Sheet 노출
3. 저장하기 버튼 클릭
4. 카테고리 선택 모달/시트 노출
5. 저장 완료

### 지도 화면의 보조 이동 버튼

우측 하단의 동그란 버튼은 **책 아이콘**으로 표시한다.  
이 버튼은 사용자를 **카테고리 관리 / TODO 관리 화면**으로 이동시키는 역할을 한다.

이 버튼의 의미는 단순 네비게이션이 아니라,  
“지도 외에 저장된 기록과 관리 영역으로 이동한다”는 의미를 가진다.

---

## 4-2. 카테고리 관리 화면

이 화면에서는 사용자가 자신이 만든 카테고리와 그 안의 장소를 볼 수 있다.

구성 예시:

- 상단 제목: My Categories
- 카테고리 리스트
- 각 카테고리별 장소 수 표시
- 카테고리 생성 버튼
- 카테고리 클릭 시 내부 장소 목록 진입

카테고리 상세 화면에서는 다음 정보를 볼 수 있다.

- 장소 이름
- 방문 상태
- 방문 날짜
- TODO 연결 여부 (선택)

예시:

- 카페 (12)
- 숙소 (3)
- 액티비티 (5)

---

## 4-3. TODO 관리 화면

TODO 관리 화면은 지도와 별개의 생산성/계획 영역이다.

구성 예시:

- 상단 제목: My Todos
- TODO 리스트
- 완료/미완료 상태
- 연관 장소 표시
- 새 TODO 추가 버튼

예시:

- [ ] 체크인 시간 확인하기 — 파르나스 호텔
- [x] 예약 문의하기 — Cafe Onion
- [ ] 돗자리 준비하기 — 한강공원

---

## 4-4. 관리 화면에서 지도 복귀

카테고리 관리 화면이나 TODO 관리 화면에서는  
우측 하단의 동그란 버튼이 **지도 아이콘**으로 표시된다.

이 버튼을 누르면 사용자는 다시 지도 화면으로 복귀한다.

즉, 화면 전환 구조는 다음과 같이 대칭적으로 설계한다.

- 지도 화면 → 책 아이콘 버튼 → 관리 영역 진입
- 관리 영역 → 지도 아이콘 버튼 → 지도 화면 복귀

이 방식은 하단 네비게이션 바 없이도  
두 성격이 다른 영역(지도 / 관리)을 자연스럽게 오갈 수 있게 해준다.

---

## 5. UX 설계 원칙

### 5-1. 지도와 관리 영역은 “다른 기능”이 아니라 “다른 보기 방식”

지도는 탐색과 저장의 시작점이다.  
관리 화면은 저장된 데이터와 TODO를 정리하는 공간이다.

즉:

- 지도 = 탐색 / 저장 중심
- 관리 = 분류 / 확인 / 계획 중심

이 둘은 별개 서비스가 아니라, 같은 데이터의 다른 접근 방식으로 본다.

---

### 5-2. 카테고리 생성은 저장 흐름 안에서 유도

카테고리를 독립 메뉴에서만 만들게 하면 사용자가 흐름을 끊고 이동해야 한다.  
그래서 장소 저장 과정에서 “새 카테고리 만들기”를 함께 제공하는 것이 중요하다.

예시:

- 저장할 카테고리 선택
- - 새 카테고리 만들기

이렇게 하면 사용자는 자연스럽게 카테고리 기능을 익히게 된다.

---

### 5-3. TODO는 별도 섹션이지만 장소와 연결 가능해야 함

TODO는 지도와 직접적인 기능은 아니지만,  
데이트 서비스에서는 실제 행동과 준비물, 계획이 매우 중요하다.

따라서 TODO는 독립적으로 관리하되,  
장소와 연결할 수 있도록 만들어 서비스의 일관성을 유지한다.

---

## 6. 데이터 구조 초안

### Category

- id
- name
- createdAt

### Place

- id
- externalPlaceId
- name
- address
- lat
- lng
- categoryId
- visitStatus
- visitedAt

visitStatus 예시:

- not_visited
- visited

### Todo

- id
- title
- isDone
- linkedPlaceId (nullable)
- createdAt

초기 MVP에서는 구조를 단순하게 유지한다.  
이후 확장 시에는 다음도 고려할 수 있다.

- Place와 Category 다대다 구조
- Todo와 Place 다대다 구조
- 방문 메모 / 사진 추가
- Timeline / Calendar View 확장

---

## 7. 이번 단계의 핵심 가치

이번 단계가 완료되면 Mapmory는 단순히 장소를 찾는 서비스가 아니라,  
**장소를 저장하고, 분류하고, 방문 여부를 관리하고, 관련 TODO까지 연결하는 서비스**가 된다.

즉, 서비스의 성격이 다음처럼 확장된다.

- 1단계: 지도 탐색 서비스
- 2단계: 장소 아카이브 + 계획 관리 서비스

---

## 8. 향후 확장 방향

이번 단계에서 쌓이는 데이터는 이후 기능으로 자연스럽게 확장될 수 있다.

### 확장 가능 기능

- 방문 기록 Timeline
- Calendar View
- 장소별 일기/메모
- 사진 기록
- 카테고리별 통계
- 완료한 TODO 기반 회고

즉, 지금 설계하는 카테고리 / 방문 날짜 / TODO 연결 구조는  
이후 “지도 + 기록 + 계획” 서비스로 발전하는 기반이 된다.
